# 移动App工厂 - 项目规则

> 所有模型、所有Skill、所有工作流必须遵守的硬性约束。

## 一、架构规则（不可违背）

### R1. 模型无关优先
- 工作流的"骨架"必须以文件形式存储（Skill定义 + Python脚本）
- 任何模型（GLM/Claude/GPT/Gemini等）替换后，工作流必须仍能跑通
- 禁止在Skill里硬编码某模型的特殊能力（如特定function calling格式）

### R2. 云端优先
- 所有重资源任务（构建、测试、打包）必须在TRAE云沙箱执行
- 本地仅负责：代码编辑、轻量预览、用户交互
- 禁止依赖本地Android SDK/Java环境

### R3. 使用GitHub
- 代码托管：GitHub仓库（zeus-666641/-APP-）
- 版本控制：本地Git + 每次commit自动推送到GitHub
- 工作模式：云端沙箱本地commit → 自动push → 跨设备/跨会话可同步
- CI/CD：仍用TRAE Hook + Schedule + 云沙箱，不用GitHub Actions（GitHub仅作代码托管）

### R4. 仅Android
- 构建目标：APK
- 最低API：Android 7.0（API 24）
- 目标API：Android 14（API 34）
- 不考虑iOS（避免macOS云环境成本）

## 二、技术栈规则

### R5. 语言与框架
- 开发语言：Python 3.10+
- UI框架：Flet（基于Flutter引擎）
- 包管理：uv（速度优于pip）
- 测试：pytest + flet.testing
- 代码风格：black + ruff

### R6. 数据存储
- 应用内数据：JSON或SQLite（视复杂度）
- 配置文件：YAML或TOML
- 禁止：二进制格式、需要额外依赖的格式

### R7. 资源文件
- 图片/图标：优先使用Material Icons（Flet内置）
- 字体：使用系统默认或Flet内置
- 音频：必要时用ogg格式
- 禁止：依赖外部资源文件，所有资源必须代码生成或网络加载

## 三、工作流规则

### R8. 节点契约
- 每个Skill节点必须明确定义：**输入**、**输出**、**失败处理**
- 节点间通过文件传递数据（避免上下文丢失）
- 节点输出必须可序列化（JSON/YAML）

### R9. 错误处理
- 任何节点失败必须写入 `操作日志.md`
- 失败后可重试3次，仍失败则暂停并通知用户
- 禁止"静默失败"

### R10. 决策权
- 技术选型、架构设计：以权威方法为主，用户意见为辅
- 产品方向、商业模式、UI风格：用户决策
- 关键决策必须记录到 `记忆/决策.md`

## 四、安全与合规

### R11. 沙箱隔离
- 所有外部命令在TRAE沙箱内执行
- 高风险命令（rm -rf、sudo等）必须用户确认
- 禁止访问宿主机文件系统（除工作目录）

### R12. 敏感信息
- API Key、密码等：用环境变量，不写入代码
- 凭证文件：放在 `.trae/` 目录，不入库
- 日志：禁止打印敏感信息

## 五、记忆与知识管理

### R13. 三类记忆
1. **决策**（`记忆/决策.md`）：重大选型、方向变更
2. **经验**（`记忆/经验.md`）：踩坑、最佳实践
3. **项目记忆**（`记忆/项目记忆.md`）：长期上下文

### R14. 操作日志
- 每次工作流执行追加一条记录到 `操作日志.md`
- 格式：`[时间] [阶段] [节点] [结果] [耗时]`
- 失败记录必须包含错误堆栈

## 六、文件组织

### R15. 文件大小限制
- 单个.py文件不超过1000行
- 超过5000行必须拆分到子文件夹
- Skill定义文件不超过200行

### R16. 命名规范
- Python文件：snake_case
- Skill文件：kebab-case.md
- 目录：中文或英文均可，但同一项目内统一
- 类：PascalCase；函数/变量：snake_case

## 七、版本与发布

### R17. 版本号
- 语义化版本：MAJOR.MINOR.PATCH
- 每次发布递增PATCH，功能变更递增MINOR，破坏性变更递增MAJOR

### R18. 发布前必检
- 所有测试通过
- 性能达标（冷启动<3s，包体积<50MB）
- 无已知P0/P1 bug
- 用户手动确认发布
