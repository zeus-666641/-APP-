## 1.1 市场调研结论

### 1.1.1 关键词扩展
对单人开发者（Python 不到二级 + 本地低配 + 无后端）适合的5大方向：
- 实用工具类（计算器/换算/手电筒）
- 本地内容展示类（菜谱/诗歌/绕口令）
- 单机小游戏（2048/翻牌配对/井字棋）
- 学习辅助卡片类（闪卡/单词）
- 个人记录类（习惯/记账/日记）

2026 年轻量增长方向：习惯追踪、订阅追踪、AI Voice Journaling、Niche Habit Tracker。

### 1.1.3 跨应用用户痛点（1~3 星评论汇总）

| # | 痛点 | 出现频次 | 根因 |
|---|---|---|---|
| P1 | streak 重装/同步/服务器宕机后丢失 | 极高 | 数据存云端或仅本地无备份 |
| P2 | 流程中插入 paywall（非 onboarding 时显示） | 高 | 商业化设计 |
| P3 | 习惯数量限制（3/5/12 个） | 高 | 免费层缩水 |
| P4 | 跨日打卡（凌晨 1 点）算昨日断流 | 中 | 时区/日期切换逻辑硬编码 |
| P5 | 习惯无法拖动排序 | 中 | UI 灵活度差 |
| P6 | 时区/DST bug（夏令时提醒错乱） | 中 | 时区处理不当 |
| P7 | 弹窗广告/特惠骚扰 | 中 | 商业化设计 |
| P8 | 通知要么骚扰要么沉默 | 中 | 通知策略粗糙 |
| P9 | UI 老旧、不美观 | 中 | 维护乏力 |
| P10 | 无法按特定星期几提醒 | 低 | 调度模型不灵活 |
| P11 | 游戏化疲劳（2 周后失去新鲜感） | 中 | Habitica 通病 |
| P12 | 数据导出能力差 | 低 | 商业锁定 |

### 1.1.4 趋势判断（2026）

- **隐私优先 / 离线优先** 成主流（innowise、dianapps、gmtasoftware 一致观点）；用户对云同步反感上升
- **跨平台** 是默认期望；但 Loop、Streaks 等成功 App 都只做单平台
- **轻量化** 是反主流机会：市场过度内卷 AI/游戏化，反而「干净、零打扰、本地」是空白
- **中文市场盲点**：Loop/HabitBox 等热门均英文优先，中文 UI+中文习惯模板（早睡/喝水/读书/八段锦）是机会
- **习惯追踪市场** 2026 仍 4.2 亿活跃用户，未饱和

## 1.2 方案生成

### 候选 A（优化型）：极简习惯追踪 · 中文优先
- **核心思路**：复制 Loop 核心功能 + 修复全部痛点 + 中文优先
- **目标用户**：中文用户，已用过 Loop 但抱怨 UI 老旧/无中文/无热图
- **MVP 功能**（5 项）：
  1. 创建/编辑/删除习惯（名称/图标/颜色/频率）
  2. 每日打卡（一次点击）
  3. Streak 计数 + 日历热图（GitHub 风格）
  4. 提醒通知（按特定星期几/时间）
  5. 本地 JSON 备份 + 一键导出 CSV
- **差异化**：解决 P1/P3/P4/P5/P6/P10 全部痛点；中文 UI 优先；零广告零 paywall
- **风险**：与 Loop 高度同质，差异化不足以撬动用户

### 候选 B（细分型）：晨晚仪式习惯追踪 · 仪式感设计
- **核心思路**：把习惯追踪限定在「早晨 30 分钟 + 晚上 30 分钟」两个仪式时段
- **目标用户**：希望早起/早睡/建立早晚仪式的年轻职场人
- **MVP 功能**（5 项）：
  1. 晨仪式 / 晚仪式两个时段容器
  2. 每个仪式内顺序化习惯链（喝水→冥想→日记→读书）
  3. 仪式进度条 + 完成度百分比
  4. 仪式开始/结束提醒（带个性化文案）
  5. 30 天仪式养成报告
- **差异化**：垂直细分「仪式」场景，Loop/HabitBox 都不区分时段；情绪化文案 + 治愈系配色
- **风险**：用户面更窄；仪式感设计需要文案/视觉反复打磨

### 候选 C（创新型）：场景触发习惯追踪 · 「锚点-行为」链
- **核心思路**：基于 BJ Fogg 行为模型「锚点（已有动作）→ 新习惯」绑定
- **目标用户**：习惯养成失败多次、需要"挂钩"策略的用户
- **MVP 功能**（5 项）：
  1. 创建「锚点-行为」对（例：刷牙后→深蹲 10 次）
  2. 锚点检测（手动确认 / 后期接 NFC/二维码触发）
  3. 行为完成打卡 + 连击天数
  4. 一周锚点效率报告（哪些锚点成功率最高）
  5. 锚点推荐库（晨起/饭后/通勤/睡前等高频锚点）
- **差异化**：锚点-行为链是习惯科学核心方法论，但市面 App 都不直接做；理论背书强
- **风险**：用户教育成本高；MVP 阶段只能手动确认锚点，体验差

### 评分矩阵（市场25% + 竞争20% + 技术25% + 变现20% + 痛点10%）

| 维度 | 权重 | A 优化型 | B 细分型 | C 创新型 |
|---|---|---|---|---|
| 市场需求 | 25% | 9（习惯追踪刚需） | 6（仪式细分市场窄） | 5（理论新，需教育） |
| 竞争烈度（分越低越激烈） | 20% | 5（红海） | 8（蓝海） | 9（蓝海） |
| 技术可行（Python+Flet+单人） | 25% | 9（标准 CRUD+列表） | 8（流程稍复杂） | 5（锚点检测需 NFC/扫码） |
| 变现潜力 | 20% | 5（同质化难定价） | 8（细分付费意愿强） | 7（理论背书可付费） |
| 痛点解决度 | 10% | 8（解 P1/3/4/5/6/10） | 6（部分痛点） | 5（少解痛点） |
| **加权总分** | 100% | **7.30** | **7.20** | **6.10** |

A、B 极接近。A 风险低、技术稳，B 差异化强、变现好。
