## 2026-07-20 移动App工厂总体架构选型
- **背景**：用户希望AI接管99%移动App开发工作，电脑配置差，Python不到二级
- **选项**：
  1. React Native + GitHub Actions（需JavaScript能力，淘汰）
  2. Flutter + Codemagic（需Dart能力，淘汰）
  3. Flet + TRAE云沙箱（Python原生，云端构建）
  4. Kivy + Buildozer（生态老旧，淘汰）
- **决策**：选 Flet + TRAE云沙箱
- **理由**：
  - Flet基于Flutter引擎，UI质量等同原生Flutter
  - Python编写，匹配用户唯一掌握的语言
  - TRAE云沙箱Linux容器可装Android SDK，编译APK不占本地资源
- **影响**：整个工作流基于Python+Flet+TRAE云沙箱构建
- **状态**：生效（2026-07-20更新：代码托管由"不用GitHub"改为"GitHub仓库 zeus-666641/-APP-"，由决策记录#20260720-github-pivot替代）

## 2026-07-20 工作流形态：扣子式编排
- **背景**：用户要求"像扣子一样的工作流，让任意模型都可以复用"
- **选项**：
  1. 纯对话式（依赖模型记忆，不可复用）
  2. 纯脚本式（无智能，不可应对变化）
  3. 骨架+执行分离（Skill定义+脚本固化，模型作为执行器可热插拔）
- **决策**：选 骨架+执行分离
- **理由**：
  - 工作流逻辑以文件存储（Skill定义+Python脚本），模型无关
  - 提示词模板化，任何模型可读
  - 确定性逻辑脚本化，避免模型不稳定
- **影响**：建立 `skills/`、`scripts/`、`记忆/` 三层结构
- **状态**：生效

## 2026-07-20 目标平台：仅Android
- **背景**：用户明确目标平台为仅Android
- **选项**：仅Android / 仅iOS / 双平台 / Web优先
- **决策**：仅Android
- **理由**：
  - iOS构建必须macOS，云macOS成本高
  - Android可在Linux云沙箱构建，零额外成本
  - 国内市场Android份额更高
- **影响**：构建流水线只针对APK，不考虑IPA
- **状态**：生效

## 2026-07-20 APK构建方式：TRAE云沙箱
- **背景**：用户电脑配置差，本地无法编译APK
- **选项**：
  1. Flet Cloud Pack（需付费Pro账号）
  2. TRAE云沙箱构建（免费，Linux容器装Android SDK）
  3. 本地ADB构建（电脑扛不住）
- **决策**：TRAE云沙箱构建
- **理由**：
  - 免费使用TRAE内置能力
  - 完全隔离，不影响本地
  - 可通过 `flet build apk` 命令直接生成APK
- **影响**：flet-cloud-builder Skill的核心动作是在云沙箱执行构建命令
- **状态**：生效

## 2026-07-20 代码托管改为GitHub（#20260720-github-pivot）
- **背景**：原 R3 规则"不用GitHub"导致跨会话/跨设备同步只能靠手动网页上传，效率低；用户已在 GitHub 初始化 `zeus-666641/-APP-` 仓库
- **选项**：
  1. 继续不用GitHub（手动上传，慢且易错）
  2. 改用GitHub + PAT自动push（云端沙箱commit后自动同步）
  3. 改用其他Git托管（Gitee/GitLab，多平台维护成本）
- **决策**：选 2，使用GitHub + PAT
- **理由**：
  - GitHub 已有仓库基础
  - PAT 可在沙箱中执行 git push，无需交互输入
  - CI/CD 仍用 TRAE 云沙箱，GitHub 仅作代码托管
- **影响**：
  - 规则.md R3 由"不用GitHub"改为"使用GitHub"
  - 项目记忆.md "代码托管"字段更新
  - Git方案.md "备份策略（不用GitHub）"章节作废
  - README.md "代码托管"行更新
  - 沙箱 git remote URL 嵌入 PAT（仅本沙箱可用，不入库）
  - 项目4核心编码在独立分支 `project-4-core-coding` 进行，避免污染 main
- **状态**：生效
