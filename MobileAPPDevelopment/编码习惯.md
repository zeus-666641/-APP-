# 移动App工厂 - 编码习惯

> 所有Skill和模型生成代码时必须遵守。基于Flet + Python栈。

## 一、Flet代码规范

### F1. 单文件入口
- 每个App的入口固定为 `main.py`
- 入口仅负责启动，业务逻辑分散到 `views/`、`services/`、`models/`
- 示例：
```python
import flet as ft
from views.home import HomeView

def main(page: ft.Page):
    page.title = "MyApp"
    page.add(HomeView())

ft.app(target=main, view=ft.AppView.WEB_BROWSER)  # 开发期；发布用NATIVE
```

### F2. 视图分离
- `views/`：UI组件，只做展示和事件转发
- `services/`：业务逻辑，可单测
- `models/`：数据结构，dataclass或pydantic
- `controllers/`：连接views和services

### F3. 组件可复用
- 重复使用的UI块抽成组件，放在 `views/components/`
- 组件参数用dataclass封装，避免长参数列表

### F4. 路由管理
- 用 `page.go("/path")` 路由，不用 `page.controls.clear() + add()`
- 路由表集中在 `routes.py`

### F5. 异步优先
- IO操作（网络、文件、数据库）必须用async
- Flet的 `page.run_task()` 用于后台任务
- 禁止在UI线程跑长任务

### F6. 设计文件复用（Q53 决策）
- 任何 UI 层编码前必须先 Read 设计文件：
  - `docs/ui_design/ui_design_spec.md`（Design Tokens + 组件规范 + 页面规范 + 交互规范）
  - `docs/ui_design/{app}_ui.html`（HTML 高保真原型）
- 颜色/字号/间距/圆角：仅从 spec 的 Design Tokens 中取值，禁止散落硬编码
- 组件命名与 spec 保持一致：TaskCard / StepCard / LogCard / section-title / AppBar / NavigationBar / Dialog / Drawer
- 同类卡片复用同一布局骨架（边距/padding/border 一致），仅内容不同
- 卡片左下角统一放删除按钮（与 LogCard 一致，Q49 决策）
- 抽屉关闭：点击遮罩 + 取消按钮双关闭（Q52 决策）；关闭时清除 `overlay.content` 引用避免黑屏
- 修改设计文件后必须同步代码（见 R21）

## 二、Python代码规范

### P1. 类型注解
- 所有函数签名必须带类型注解
- 公共API必须有docstring（Google风格）

```python
def fetch_user(user_id: str) -> User | None:
    """获取用户信息
    
    Args:
        user_id: 用户唯一标识
    
    Returns:
        User对象，不存在返回None
    """
```

### P2. 异常处理
- 不允许裸 `except:`，必须捕获具体异常
- 业务异常继承自 `AppError` 基类
- 边界异常（网络、文件）转成业务异常抛出

### P3. 配置管理
- 配置用 `pydantic-settings` 或简单dataclass
- 禁止硬编码：URL、API Key、路径、魔法数字
- 配置文件放 `config/`，按环境分（dev/prod）

### P4. 日志
- 用 `logging` 模块，不用print
- 日志级别：DEBUG/INFO/WARNING/ERROR
- 敏感信息脱敏后再记录

### P5. 测试
- 测试文件命名：`test_{module}.py`
- 测试函数：`test_{行为}_{条件}_{预期}`
- 用fixtures复用测试数据
- 目标覆盖率：核心逻辑 >80%，UI >50%

## 三、性能习惯

### PER1. 懒加载
- 大型组件用 `Lazy` 加载
- 图片用缩略图，按需加载原图
- 列表用 `ListView.builder` 而非 `Column`

### PER2. 内存
- 不用的对象显式置空
- 大文件用流式读取
- 缓存用 `functools.lru_cache`

### PER3. 启动优化
- 启动时只加载首屏需要的资源
- 后续页面懒加载
- Splash screen不超过2秒

## 四、文件组织

### ORG1. 目录结构（每个App）
```
apps/{app_name}/
├── main.py                # 入口
├── pyproject.toml         # 依赖
├── config/
│   ├── settings.py
│   └── settings.prod.py
├── views/
│   ├── home.py
│   ├── settings.py
│   └── components/
├── services/
│   ├── api.py
│   └── storage.py
├── models/
│   └── user.py
├── controllers/
├── routes.py
├── tests/
│   ├── test_services/
│   └── test_views/
└── resources/             # 仅必要时
```

### ORG2. 文件大小
- 单文件不超过500行
- 超过则按职责拆分

### ORG3. 命名
- 模块文件：snake_case
- 类：PascalCase
- 函数/变量：snake_case
- 常量：UPPER_SNAKE
- 私有：前缀_

## 五、提交规范

### COM1. 提交粒度
- 一次提交一个完整功能或修复
- 不混合多个无关变更

### COM2. 提交信息
```
<type>(<scope>): <subject>

<body>
```
- type：feat/fix/refactor/test/docs/chore
- scope：模块名
- subject：简短描述

### COM3. 测试先行
- 新功能先写测试
- 修bug先复现bug的测试
- 测试不过不提交

## 六、移动端特殊习惯

### MOB1. 响应式布局
- 所有页面在手机和平板上都正常显示
- 用 `ResponsiveRow` 或媒体查询

### MOB2. 离线友好
- 网络请求失败有缓存兜底
- 关键操作可离线完成
- 离线状态有明确提示

### MOB3. 触控优化
- 按钮最小点击区域48x48dp
- 重要按钮在拇指可达区
- 避免误触：危险操作二次确认

### MOB4. 权限最小化
- 只申请必要权限
- 权限申请时机：用到时才申请
- 拒绝权限有降级方案

### MOB5. 国际化
- 文案集中在 `i18n/`，不硬编码
- 默认支持中文和英文
- 数字、日期、货币按locale格式化

## 七、每次编码后检查清单（24 项）

> 每次完成编码工作后，按本清单逐项自检。质量是第一优先级，不可跳过。
> 用户原文中的"游戏"在此项目语境下指"App/项目"（step_recorder 等所有应用通用）。

### CHK1. 每次检查
- 每次编码完成都要执行下面的检查清单，不可省略

### CHK2. 检查文件是否需要更新
- 检查是否需要更新文档（PRD、决策、骨架、记忆）
- 检查是否需要更新 README、CHANGELOG
- 检查是否需要更新 ROADMAP.md（用户原文 POSNMAP.md，推断为 ROADMAP 路线图）

### CHK3. 检查按键反馈是否需要更新
- 按钮点击/拖拽/切换等交互是否有视觉反馈
- 长操作是否有进度提示
- 错误是否有 SnackBar/Dialog 反馈

### CHK4. 新更新的内容是否导致 bug 产生
- 是否与其他代码冲突
- 是否显示正常
- 是否正常工作
- 是否导致原有内容缺失或者错误

### CHK5. 需要存档的数据是否需要更新
- 数据模型/字段是否变更
- 存储格式（JSON/SQLite schema）是否需要迁移
- 旧数据是否兼容

### CHK6. 设置是否有需要扩展的，版本号需不需要更新
- 设置项是否需要新增/调整
- 应用版本号是否需要递增（PATCH/MINOR/MAJOR）

### CHK7. 更新内容是否合理，是否需要进行合理性检测
- 修改是否符合 PRD 与决策记录
- 是否需要合理性自检（不破坏既有约束）

### CHK8. 文件中是否存在错误，代码是否有错误解决方案
- 是否有语法错误、类型错误、运行时错误
- 是否有对应解决方案或回退方案

### CHK9. 更新内容是否重大或者丰富
- 如果是，根据现有条件往后可发展方向有哪些（记录到扩展清单，暂不实现）

### CHK10. .py 文件是否需要分类
- 是否需要新建文件来细分代码
- 是否需要新建文件夹来细分 .py 文件
- 是否需要新建文件来更好地管理代码和资源
- 是否为了方便以后更简单地修改代码
- 是否需要删除某些文件
- 文件是否太过于冗长（参考 ORG2：单文件不超过 500 行）

### CHK11. 更新内容是否有效
- 改动是否真正解决问题
- 是否存在空操作或无意义的改动

### CHK12. 更新内容是否合理
- 是否符合最小变更原则
- 是否过度设计

### CHK13. 更新数据是否合理，运行检查文件
- 数据结构变更是否合理
- 是否运行了测试（pytest）验证

### CHK14. 是否需要更新 ROADMAP.md（用户原文 POSNMAP/ROSNMAP.md）
- 更新内容是否需要记录到路线图
- 路线图状态是否需要刷新

### CHK15. 是否存在屎山代码、无效代码、低性能代码
- 重复代码块是否抽取
- 死代码是否删除
- 低性能写法（N+1 查询、O(n²) 循环、不必要的全量重渲染）是否优化

### CHK16. 是否需要更新 requirements 文件
- 新增依赖是否已加入 pyproject.toml / requirements.txt
- 移除的依赖是否清理

### CHK17. 是否需要更新编码习惯文件
- 本次踩坑是否需要沉淀为新规范
- 是否有新模式需要补到本文件

### CHK18. 是否需要更新 .gitignore 文件
- 新生成的产物（缓存、临时文件）是否需要忽略
- 敏感文件（.env、凭证）是否已忽略

### CHK19. 是否需要压缩上下文来释放上下文空间
- 当前对话是否过长，是否需要总结压缩
- 是否有冗余信息可以归档到记忆文件后从对话移除

### CHK20. 是否有未完成的任务需要继续再来处理新任务
- 是否有遗留待办，应在新任务前完成
- 是否需要回到旧任务而非开新任务

### CHK21. 任务是否需要调用截图工具来获取任务的进行情况或者程序的运行情况
- UI 改动是否需要截图验证
- 是否需要预览来确认显示正常

### CHK22. 是否有你不确定不理解的地方、是否有用户讲的不详细不对的地方
- 这些情况应当提问
- 不确定的地方不可擅自决策，必须先问

### CHK23. 继续编码
- 检查完成后继续推进下一项任务
- 质量优先，不贪快

### CHK24. UI 一致性自检（Q53 决策）
- 是否先 Read 了 `docs/ui_design/ui_design_spec.md` 和 `{app}_ui.html` 才开始 UI 层编码
- 颜色 HEX 是否全部来自 Design Tokens（搜代码中是否有散落的 `#xxxxxx` 不在 token 表）
- 字号是否使用了 spec 之外的档位（如 13、15、17 等奇数）
- 间距是否使用了 spec 之外的档位（如 5、7、9 等奇数）
- 圆角是否使用了 spec 之外的档位
- 同类卡片（TaskCard/StepCard/LogCard）的 padding/border/border_radius 是否一致
- 删除按钮是否在卡片左下角（与 LogCard 对齐）
- 可点控件命中区是否 ≥48×48dp
- 危险操作（删除/重置）是否二次确认（modal AlertDialog）
- 抽屉关闭是否支持「点击遮罩 + 取消按钮」双关闭（Q52）
- 抽屉关闭后是否清除了 `overlay.content` 引用（避免黑屏）
- 修改设计文件后是否同步更新了 `ui_design_spec.md` 和 HTML 原型（见 R21）

### 检查执行约定
- **执行时机**：每次完成一个功能/修复/重构后立即自检
- **执行方式**：逐项对照，发现的问题记录到当前任务的待办或提问清单
- **遇到提问项**：按用户约定"遇到要提问的放到最后"，集中提问
- **不可跳过**：哪怕改动很小，也要走完 24 项（可以快速过）
