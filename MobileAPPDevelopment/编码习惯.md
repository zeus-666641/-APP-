# 移动App工厂 - 编码习惯

> 所有Skill和模型生成代码时必须遵守。基于Flet + Python栈。

## 一、Flet代码规范

### F1. 单文件入口
- 每个App的入口固定为 `main.py`
- 入口仅负责启动，业务逻辑分散到 `views/`、`services/`、`models/`
- 示例：
```python
import flet as ft
from views.home import HomeView

def main(page: ft.Page):
    page.title = "MyApp"
    page.add(HomeView())

ft.app(target=main, view=ft.AppView.WEB_BROWSER)  # 开发期；发布用NATIVE
```

### F2. 视图分离
- `views/`：UI组件，只做展示和事件转发
- `services/`：业务逻辑，可单测
- `models/`：数据结构，dataclass或pydantic
- `controllers/`：连接views和services

### F3. 组件可复用
- 重复使用的UI块抽成组件，放在 `views/components/`
- 组件参数用dataclass封装，避免长参数列表

### F4. 路由管理
- 用 `page.go("/path")` 路由，不用 `page.controls.clear() + add()`
- 路由表集中在 `routes.py`

### F5. 异步优先
- IO操作（网络、文件、数据库）必须用async
- Flet的 `page.run_task()` 用于后台任务
- 禁止在UI线程跑长任务

## 二、Python代码规范

### P1. 类型注解
- 所有函数签名必须带类型注解
- 公共API必须有docstring（Google风格）

```python
def fetch_user(user_id: str) -> User | None:
    """获取用户信息
    
    Args:
        user_id: 用户唯一标识
    
    Returns:
        User对象，不存在返回None
    """
```

### P2. 异常处理
- 不允许裸 `except:`，必须捕获具体异常
- 业务异常继承自 `AppError` 基类
- 边界异常（网络、文件）转成业务异常抛出

### P3. 配置管理
- 配置用 `pydantic-settings` 或简单dataclass
- 禁止硬编码：URL、API Key、路径、魔法数字
- 配置文件放 `config/`，按环境分（dev/prod）

### P4. 日志
- 用 `logging` 模块，不用print
- 日志级别：DEBUG/INFO/WARNING/ERROR
- 敏感信息脱敏后再记录

### P5. 测试
- 测试文件命名：`test_{module}.py`
- 测试函数：`test_{行为}_{条件}_{预期}`
- 用fixtures复用测试数据
- 目标覆盖率：核心逻辑 >80%，UI >50%

## 三、性能习惯

### PER1. 懒加载
- 大型组件用 `Lazy` 加载
- 图片用缩略图，按需加载原图
- 列表用 `ListView.builder` 而非 `Column`

### PER2. 内存
- 不用的对象显式置空
- 大文件用流式读取
- 缓存用 `functools.lru_cache`

### PER3. 启动优化
- 启动时只加载首屏需要的资源
- 后续页面懒加载
- Splash screen不超过2秒

## 四、文件组织

### ORG1. 目录结构（每个App）
```
apps/{app_name}/
├── main.py                # 入口
├── pyproject.toml         # 依赖
├── config/
│   ├── settings.py
│   └── settings.prod.py
├── views/
│   ├── home.py
│   ├── settings.py
│   └── components/
├── services/
│   ├── api.py
│   └── storage.py
├── models/
│   └── user.py
├── controllers/
├── routes.py
├── tests/
│   ├── test_services/
│   └── test_views/
└── resources/             # 仅必要时
```

### ORG2. 文件大小
- 单文件不超过500行
- 超过则按职责拆分

### ORG3. 命名
- 模块文件：snake_case
- 类：PascalCase
- 函数/变量：snake_case
- 常量：UPPER_SNAKE
- 私有：前缀_

## 五、提交规范

### COM1. 提交粒度
- 一次提交一个完整功能或修复
- 不混合多个无关变更

### COM2. 提交信息
```
<type>(<scope>): <subject>

<body>
```
- type：feat/fix/refactor/test/docs/chore
- scope：模块名
- subject：简短描述

### COM3. 测试先行
- 新功能先写测试
- 修bug先复现bug的测试
- 测试不过不提交

## 六、移动端特殊习惯

### MOB1. 响应式布局
- 所有页面在手机和平板上都正常显示
- 用 `ResponsiveRow` 或媒体查询

### MOB2. 离线友好
- 网络请求失败有缓存兜底
- 关键操作可离线完成
- 离线状态有明确提示

### MOB3. 触控优化
- 按钮最小点击区域48x48dp
- 重要按钮在拇指可达区
- 避免误触：危险操作二次确认

### MOB4. 权限最小化
- 只申请必要权限
- 权限申请时机：用到时才申请
- 拒绝权限有降级方案

### MOB5. 国际化
- 文案集中在 `i18n/`，不硬编码
- 默认支持中文和英文
- 数字、日期、货币按locale格式化
