# 移动App完整制作流程清单

> 从0到1做一个Android App的全流程拆解。每个项目（环节）→流程→步骤→设计→编码要点。

## 总览：7大项目（环节）

```
项目1：选题与立项
项目2：产品规划
项目3：架构与脚手架
项目4：核心编码
项目5：质量保障
项目6：打包封装
项目7：发布与运营
```

每个项目包含若干流程，每个流程包含若干步骤。

---

## 项目1：选题与立项

**目标**：决定"做什么App"，输出可立项的方案。

### 流程1.1：市场调研
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 1.1.1 关键词扩展 | 从用户兴趣扩展到10+相关关键词 | WebSearch：`{关键词} App 排行` |
| 1.1.2 竞品收集 | 找出前10个竞品，记录名称/评分/下载量 | WebSearch + 应用商店页面 |
| 1.1.3 用户痛点挖掘 | 从竞品差评和论坛抱怨提炼痛点 | WebSearch：`{关键词} 吐槽 痛点` |
| 1.1.4 趋势判断 | 判断赛道是上升/平稳/衰退 | WebSearch：`{关键词} 趋势 2026` |

### 流程1.2：方案生成
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 1.2.1 候选方案设计 | 生成3个差异化方案（优化型/细分型/创新型） | 模型生成 |
| 1.2.2 MVP范围界定 | 每个方案圈定3-5个核心功能 | 模型生成 |
| 1.2.3 评分矩阵 | 加权打分：市场25%+竞争20%+技术25%+变现20%+痛点10% | Python脚本计算 |

### 流程1.3：立项决策
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 1.3.1 用户选择 | 让用户从3个方案中选1个 | AskUserQuestion |
| 1.3.2 方案归档 | 写入决策记录、生成初稿PRD | Write到`docs/prd/draft.md` |

**产出**：`docs/prd/draft.md`、`记忆/选题调研_{时间}.json`

## 项目2：产品规划

**目标**：把方案转成可执行的开发计划。

### 流程2.1：PRD撰写
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 2.1.1 功能清单 | 列出所有功能，标注MVP/后续 | 模型生成，写入`docs/prd/{app}_prd.md` |
| 2.1.2 用户流程图 | 描述主要用户路径（注册→使用→付费） | Mermaid语法 |
| 2.1.3 数据模型 | 定义核心实体（User/Item/Order等） | Python dataclass |
| 2.1.4 UI草图 | 每个页面的文字描述+布局结构 | 模型生成 |

### 流程2.2：架构设计
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 2.2.1 目录结构 | views/services/models/controllers分层 | 见编码习惯.md ORG1 |
| 2.2.2 模块依赖 | 明确模块间调用关系 | 文字描述 |
| 2.2.3 技术选型 | 选定第三方库（网络/存储/状态管理） | 模型决策 |

### 流程2.3：任务拆解
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 2.3.1 任务粒度 | 每个任务<4小时，独立可测 | 模型生成 |
| 2.3.2 任务排序 | 按依赖关系排序 | 拓扑排序 |
| 2.3.3 任务清单 | 写入`docs/tasks.md`，含工时估算 | Write |

**产出**：`docs/prd/{app}_prd.md`、`docs/architecture.md`、`docs/tasks.md`

## 项目3：架构与脚手架

**目标**：生成可运行的项目骨架。

### 流程3.1：模板准备
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 3.1.1 模板目录 | `templates/flet-app-template/` 完整Flet项目 | 预先创建 |
| 3.1.2 模板配置 | `pyproject.toml`、`.gitignore`、`README.md` | 静态文件 |
| 3.1.3 占位符 | 用`{app_name}`、`{app_title}`等占位 | 字符串模板 |

### 流程3.2：脚手架生成
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 3.2.1 复制模板 | 从模板复制到`apps/{app_name}/` | Python脚本 `scripts/scaffold.py` |
| 3.2.2 替换占位符 | 用PRD信息填充模板 | Python字符串替换 |
| 3.2.3 生成路由 | 根据PRD页面清单生成`routes.py` | 模型生成 |

### 流程3.3：依赖安装
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 3.3.1 创建虚拟环境 | 用`uv venv`或`python -m venv` | RunCommand |
| 3.3.2 安装依赖 | `uv sync`或`pip install -e .` | RunCommand |
| 3.3.3 启动验证 | `python main.py` 浏览器预览 | RunCommand |

**产出**：`apps/{app_name}/` 可运行骨架

## 项目4：核心编码

**目标**：按任务清单实现所有功能。这是工作量最大的项目。

### 流程4.1：数据层
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 4.1.1 数据模型 | 用dataclass或pydantic定义实体 | `models/{entity}.py` |
| 4.1.2 存储层 | SQLite或JSON文件存储 | `services/storage.py` |
| 4.1.3 数据迁移 | 版本升级时的数据迁移逻辑 | `services/migrations.py` |
| 4.1.4 单测 | 每个模型和存储方法都要测 | `tests/test_models/` |

### 流程4.2：业务层
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 4.2.1 Service类 | 每个业务领域一个Service | `services/{domain}.py` |
| 4.2.2 异步处理 | IO操作用async/await | `async def` |
| 4.2.3 错误处理 | 业务异常继承AppError | `models/exceptions.py` |
| 4.2.4 单测 | 用pytest+pytest-asyncio | `tests/test_services/` |

### 流程4.3：UI层
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 4.3.1 页面View | 每个页面一个View类 | `views/{page}_view.py` |
| 4.3.2 可复用组件 | 按钮/列表/表单抽组件 | `views/components/` |
| 4.3.3 路由注册 | 在routes.py注册新页面 | `routes.py` |
| 4.3.4 状态管理 | 用Flet的page.state或第三方 | 模型决策 |
| 4.3.5 UI测试 | 用flet.testing或截图比对 | `tests/test_views/` |

### 流程4.4：网络层（如需）
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 4.4.1 API客户端 | 用httpx或aiohttp | `services/api.py` |
| 4.4.2 请求拦截 | 统一加token、日志、错误转换 | middleware模式 |
| 4.4.3 缓存策略 | 网络失败走缓存 | `services/cache.py` |
| 4.4.4 单测 | 用respx或pytest-httpserver mock | `tests/test_api/` |

### 流程4.5：权限与原生功能
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 4.5.1 权限声明 | 在`android/app/src/main/AndroidManifest.xml`声明 | Flet配置 |
| 4.5.2 权限申请 | 运行时用Flet的permission API | 模型生成 |
| 4.5.3 原生调用 | 相机/位置/通知等 | Flet plugins |

### 流程4.6：国际化
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 4.6.1 文案提取 | 所有UI文案移到i18n | `i18n/zh.json`、`en.json` |
| 4.6.2 翻译函数 | `t("key")`返回当前语言文案 | `services/i18n.py` |
| 4.6.3 语言切换 | 设置页提供语言选择 | `views/settings_view.py` |

**产出**：完整可运行的App代码 + 测试套件

## 项目5：质量保障

**目标**：确保App质量达标，可发布。

### 流程5.1：自动化测试
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 5.1.1 单元测试 | 覆盖率目标：核心>80% | pytest + coverage |
| 5.1.2 集成测试 | 模块间协作测试 | pytest |
| 5.1.3 UI测试 | 关键用户流程E2E | flet.testing |
| 5.1.4 性能测试 | 启动时间、内存占用 | tracemalloc + time |

### 流程5.2：代码审查
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 5.2.1 静态分析 | 用ruff检查代码风格 | `ruff check .` |
| 5.2.2 类型检查 | 用mypy检查类型 | `mypy .` |
| 5.2.3 安全扫描 | 检查硬编码密钥、SQL注入 | security-best-practices Skill |
| 5.2.4 性能审查 | 识别N+1查询、内存泄漏 | mobile-perf-auditor Skill |

### 流程5.3：优化
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 5.3.1 启动优化 | 懒加载非首屏资源 | 代码重构 |
| 5.3.2 内存优化 | 释放大对象、用流式读取 | 代码重构 |
| 5.3.3 包体积优化 | 移除未用依赖、压缩资源 | `flet build --split-per-abi` |
| 5.3.4 渲染优化 | ListView.builder、避免深嵌套 | 代码重构 |

### 流程5.4：用户验收
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 5.4.1 内部测试 | 自己跑通核心流程 | 手动+脚本 |
| 5.4.2 用户测试 | 找3-5个目标用户试用 | 收集反馈 |
| 5.4.3 Bug修复 | 按P0/P1/P2优先级修 | 模型生成修复 |

**产出**：测试报告、审查报告、优化后代码

## 项目6：打包封装

**目标**：生成可分发的签名APK。

### 流程6.1：构建准备
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 6.1.1 版本号确认 | 检查`pyproject.toml`版本 | Read |
| 6.1.2 应用图标 | 生成或指定图标PNG | `assets/icon.png` |
| 6.1.3 应用配置 | 包名、版本、权限 | `flet.toml`或`pyproject.toml` |

### 流程6.2：云沙箱构建
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 6.2.1 沙箱环境检查 | 确认Python、Flet、Android SDK | RunCommand |
| 6.2.2 依赖同步 | 在沙箱内`uv sync` | RunCommand |
| 6.2.3 执行构建 | `flet build apk` 长任务 | RunCommand (async) |
| 6.2.4 构建监控 | 轮询构建状态 | CheckCommandStatus |

### 流程6.3：签名
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 6.3.1 生成keystore | 首次创建，妥善保管 | `keytool` |
| 6.3.2 APK签名 | 用apksigner | RunCommand |
| 6.3.3 签名验证 | 确认签名有效 | `apksigner verify` |

### 流程6.4：产物归档
| 步骤 | 设计要点 | 编码/工具 |
|---|---|---|
| 6.4.1 复制APK | 到`releases/`目录 | Write |
| 6.4.2 生成hash | SHA256校验 | `certutil` |
| 6.4.3 构建报告 | 记录构建信息 | Write到`docs/builds/` |

**产出**：`releases/{app}_{version}.apk` + 校验文件

## 自动化优先级

| 自动化级别 | 流程 |
|---|---|
| **全自动**（AI+脚本） | 1.1调研、1.2方案、2.1PRD、3.2脚手架、4.*编码、5.1测试、6.2构建、7.3监控 |
| **半自动**（AI建议+用户确认） | 1.3立项、2.3任务、5.4验收、7.4迭代 |
| **手动**（用户操作） | 7.1账号、7.2上传 |

预计AI可接管的工作量：**约90-95%**，符合"99%目标"的接近值。剩余5-10%是必须人工的（账号注册、商店上传、用户调研等）。
